1. Descripción del laboratorio

Objetivo general

Practicar de extremo a extremo un pipeline moderno de integración continua (CI) y análisis estático de código sobre una aplicación real con backend, frontend y base de datos. Al terminar, la pareja debe ser capaz de:

  • Levantar un stack multicontenedor con docker compose.
  • Configurar un repositorio de GitHub con ramas develop y master protegidas.
  • Escribir un workflow de GitHub Actions que ejecute lint, pruebas, cobertura y análisis Sonar.
  • Interpretar un Quality Gate de SonarCloud y entender por qué falla un PR.
  • Aumentar la cobertura de pruebas hasta superar el umbral exigido.

Modalidad y duración

  • Parejas dos integrantes por entrega.
  • ~10 horas en 2-3 sesiones (incluye lectura, configuración, ejecución y redacción del informe).
  • Entregable un único PDF por pareja (ver sección Entrega).

Estructura del kit del laboratorio

El kit que recibes consta de un repositorio Git y dos carpetas overlay fuera del repositorio:

Criterios de evaluación

El laboratorio se evalúa exclusivamente con el PDF de entrega. Cada paso técnicamente verificable debe llevar:

  1. Una captura de pantalla completa (NO recortes — debe verse barra de tareas, hora, navegador completo).
  2. Una explicación escrita de mínimo 3 líneas que diga: qué hiciste, qué muestra la captura, y por qué eso demuestra que el paso se cumplió.

Hay 20 pasos verificables (lista completa en la pestaña Entrega), cada uno vale 1 punto. Si falta la captura o la explicación es trivial, el paso no suma.

Las instalaciones, la creación de cuentas de GitHub/SonarCloud y la ejecución del script de setup no requieren captura — son preparación, no parte de la evaluación.

2. Preparación del entorno

Requisitos previos

  • Windows 10/11 con permisos para instalar software (no se necesita admin total, pero sí UAC).
  • Docker Desktop con WSL2 activado y corriendo.
  • Acceso a internet (descarga de ~200 MB durante el setup).
  • Cuenta de GitHub (gratuita).
  • Cuenta de SonarCloud (gratuita, se crea con login de GitHub).

Script de setup automatizado

Desde la raíz del repo, en PowerShell:

# Modo simulación: muestra qué haría sin tocar nada
powershell -ExecutionPolicy Bypass -File .\scripts\setup-windows.ps1 -DryRun

# Ejecución real (te pedirá ENTER para confirmar)
powershell -ExecutionPolicy Bypass -File .\scripts\setup-windows.ps1

# Sin confirmación (úsalo si ya validaste con -DryRun)
powershell -ExecutionPolicy Bypass -File .\scripts\setup-windows.ps1 -NoConfirm

El script hace:

  1. Detecta y desinstala todas las versiones de Python (incluido las gestionadas por Python Install Manager de Microsoft).
  2. Borra restos huérfanos en %LOCALAPPDATA%\Python\ y limpia PATH.
  3. Instala Python 3.12.10 en scope de usuario.
  4. Instala nvm-windows si no está, y luego Node 20.11.1.
  5. Instala GitHub CLI.
  6. Verifica Docker y git.
Importante: al desinstalar Python se pierden todas las herramientas instaladas con pip (poetry, awscli, ansible, pre-commit, etc.). Si las necesitas, reinstálalas con pip install después.

4. Git Flow del laboratorio

Modelo de ramas

gitGraph commit id: "init" branch develop checkout develop commit id: "skeleton" branch feature/etapa-1 checkout feature/etapa-1 commit id: "código y tests etapa 1" checkout develop merge feature/etapa-1 tag: "PR a develop OK" checkout main merge develop tag: "PR a master FALLA (cob.<80%)" checkout develop branch feature/etapa-2 checkout feature/etapa-2 commit id: "+aumentar-cobertura" checkout develop merge feature/etapa-2 checkout main merge develop tag: "PR a master OK"

¿Por qué separar develop de master?

  • develop es la rama de integración diaria. Aquí mergeas tu trabajo y el de tu pareja con un pipeline rápido (lint + tests sin gate estricto).
  • master (o main) representa lo que iría a producción. Aquí el pipeline es estricto: cobertura ≥80 %, Quality Gate de Sonar verde, build de Docker exitoso.
  • Esta separación es la base de modelos como GitFlow y GitHub Flow: te permite romper cosas en develop sin afectar la rama protegida.

Comandos típicos

git checkout -b develop
git push -u origin develop
git checkout -b master
git push -u origin master

# Para cada feature
git checkout develop
git checkout -b feature/mi-cambio
# ... trabajas, commiteas ...
git push -u origin feature/mi-cambio
# Y luego abres PR feature/mi-cambio -> develop en GitHub

5. Teoría: Integración Continua y Entrega Continua

¿Qué es CI (Continuous Integration)?

CI es la práctica de integrar cambios al repositorio compartido frecuentemente (al menos una vez al día) y validar automáticamente que cada integración no rompa nada. Cada commit o PR dispara una pipeline que:

  1. Trae el código del repo.
  2. Instala dependencias.
  3. Ejecuta linters.
  4. Compila / construye artefactos.
  5. Ejecuta tests automatizados.
  6. Mide cobertura.
  7. Analiza calidad estática (SonarQube/SonarCloud).

Si algún paso falla, el PR se bloquea — el cambio no entra al repo hasta que el autor lo arregle.

¿Qué es CD?

Dos significados que conviene distinguir:

ConceptoQué haceCuándo se despliega
Continuous Delivery El pipeline construye un artefacto desplegable y lo deja listo en un registro/staging. Despliegue a producción manual (un click).
Continuous Deployment Igual al anterior, pero si todo el pipeline pasa, el despliegue a producción es automático. Cada commit en master verde llega a producción solo.

En este laboratorio practicamos solo CI + Continuous Delivery parcial (construimos las imágenes Docker, pero no las desplegamos).

Beneficios reales

  • Feedback rápido: sabes en minutos si un cambio rompe algo, no en semanas.
  • Calidad consistente: el repo nunca contiene código que no pasa los tests.
  • Confianza para refactorizar: los tests + el linter te avisan si rompes algo.
  • Onboarding: un dev nuevo solo necesita seguir las instrucciones del README — el pipeline le dice si su PR está bien.

Mitos comunes

"Si tengo CI, no necesito QA manual" Falso. CI valida lo que está automatizado. Las pruebas exploratorias, de usabilidad y de aceptación siguen siendo humanas. CI te libera tiempo para esas tareas que sí requieren juicio.
"CI hace mi código bueno" No. CI te obliga a tener tests, linter y análisis estático configurados, pero la calidad del código depende del equipo. CI es la red de seguridad, no el ingeniero.
"Es solo para grandes empresas" GitHub Actions es gratis para repos públicos ilimitados. Una pareja de estudiantes monta lo mismo que una empresa Fortune 500 — la diferencia es solo escala.

6. Análisis estático: linters

¿Qué es un linter?

Un linter analiza tu código sin ejecutarlo y detecta:

  • Errores reales: variables no definidas, imports rotos, returns inconsistentes.
  • Bugs probables: comparar con == en lugar de is, modificar listas durante iteración, etc.
  • Code smells: funciones demasiado largas, complejidad ciclomática alta, código muerto.
  • Estilo: indentación, naming, líneas demasiado largas.

Al correrlo en el PR garantizamos un mínimo de calidad y consistencia antes de invertir tiempo en review humano.

Ruff (Python)

Linter de Python escrito en Rust. 10× a 100× más rápido que flake8 y reemplaza a la mayoría de plugins clásicos.

Configuración en backend/pyproject.toml:

[tool.ruff]
line-length = 100
target-version = "py312"

[tool.ruff.lint]
select = ["E", "F", "I", "B", "UP"]
# E = pycodestyle errores
# F = pyflakes (imports / vars)
# I = orden de imports (isort)
# B = bugbear (bugs comunes)
# UP = pyupgrade (sintaxis nueva)

Ejecución local:

cd backend
ruff check app          # solo reporta
ruff check app --fix    # corrige lo que pueda automáticamente

ESLint (TypeScript/React)

Linter estándar del ecosistema JS. Configurado en frontend/.eslintrc.cjs:

extends: [
  "eslint:recommended",
  "plugin:@typescript-eslint/recommended",
]

Ejecución:

cd frontend
npm run lint            # --max-warnings 0 (cualquier warning falla)

Integración en el pipeline

En ci-develop.yml el lint es un job propio que corre en paralelo con los tests:

lint-backend:
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v4
    - uses: actions/setup-python@v5
      with:
        python-version: "3.12.10"
    - run: pip install -r requirements.txt
      working-directory: backend
    - run: ruff check app
      working-directory: backend

Si falla, el check del PR se pone rojo y no permite mergear (cuando hay branch protection).

8. Cobertura de código

¿Qué mide?

La cobertura es el porcentaje de tu código que es ejecutado durante los tests. Hay varios niveles:

  • Cobertura de líneas: qué % de líneas se ejecuta al menos una vez. La más usada.
  • Cobertura de ramas (branches): qué % de caminos if/else se recorren. Más estricta.
  • Cobertura de funciones: qué % de funciones se invoca alguna vez.
  • Cobertura de sentencias: similar a líneas pero contando statements.

¿Por qué 80 % y no 100 %?

  • Ley de rendimientos decrecientes: del 0 al 70 % obtienes mucha protección con poco esfuerzo. Del 90 al 100 % te toca testear código defensivo, manejo de errores que nunca pasan, etc.
  • Falsa sensación de seguridad: 100 % de cobertura no significa 100 % de correctitud. Puedes ejecutar todas las líneas con asserts incorrectos.
  • 80 % es industria estándar: Google, Spotify, ThoughtWorks usan umbrales similares en proyectos nuevos.
Importante: 80 % no es un objetivo, es un piso. Hay código crítico (autenticación, pagos, criptografía) que debería estar al 100 %, y código trivial (getters/setters, prints) que no necesita prueba.

Cobertura con pytest-cov

Comando completo:

pytest --cov=app --cov-report=term-missing --cov-report=xml --cov-fail-under=80
  • --cov=app — qué carpeta medir.
  • --cov-report=term-missing — mostrar tabla con líneas no cubiertas.
  • --cov-report=xml — generar coverage.xml (lo lee Sonar).
  • --cov-fail-under=80 — falla si la cobertura total es <80 %. Esta es la línea que rechaza el PR.

Cobertura con Vitest

Configurado en vite.config.ts:

test: {
  coverage: {
    provider: "v8",
    reporter: ["text", "lcov"],   // lcov es lo que entiende Sonar
    thresholds: {
      lines: 80, statements: 80, functions: 80, branches: 70,
    },
  },
}

Comando: npm run test:coverage

Anti-patrones de cobertura

  • Tests sin asserts: ejecutan código pero no validan nada. Suben cobertura, no calidad.
  • Tests acoplados a la implementación: si cambias un nombre interno se rompen. Prueba comportamientos, no líneas.
  • Mocks excesivos: si tu test mockea todo lo que la función toca, en realidad estás probando el mock.
  • Cobertura como meta única: 80 % de líneas malas sigue siendo malo. Combinar con Sonar (que mira complejidad, duplicación, smells) compensa.

9. SonarCloud

¿Qué hace Sonar?

SonarCloud (versión SaaS de SonarQube) es un análisis estático profundo que detecta cuatro grandes categorías:

CategoríaEjemploSeveridad
BugsNullPointerException probable, división por ceroBloqueante
VulnerabilitiesInyección SQL, secret hardcodeado, criptografía débilBloqueante
Code SmellsFunción con 50 líneas, complejidad ciclomática 25, parámetro no usadoMayor / menor
Security HotspotsUso de regex sin anchors, CORS abierto, deserializaciónA revisar humano

Además calcula:

  • Cobertura (usa los reportes que le pasas).
  • Duplicación de código (% de líneas duplicadas).
  • Maintainability Rating (A a E) basado en code smells vs total de líneas.
  • Reliability Rating y Security Rating.

El Quality Gate

Es un conjunto de condiciones que tu código debe cumplir para considerarse "aceptable". El default Sonar Way incluye:

  • Cobertura en código nuevo ≥ 80 %.
  • Duplicación en código nuevo ≤ 3 %.
  • 0 bugs en código nuevo.
  • 0 vulnerabilidades en código nuevo.
  • Maintainability Rating de A en código nuevo.
  • Security Hotspots revisados al 100 %.

Nota la palabra "código nuevo": Sonar mide solo lo que cambiaste en el PR. Esto evita que proyectos legacy con deudas grandes queden bloqueados — la regla es "no empeores lo que tienes".

Configuración paso a paso

  1. Ir a sonarcloud.io y hacer login con GitHub.
  2. Click en + arriba a la derecha → Analyze new project.
  3. Importar el repo del laboratorio (ya debe estar público).
  4. En el wizard elegir "With GitHub Actions".
  5. Copiar el SONAR_TOKEN que aparece. Solo se muestra una vez.
  6. En GitHub: Settings → Secrets and variables → Actions → New repository secret. Nombre: SONAR_TOKEN, valor: el token copiado.
  7. Anotar Organization y Project Key de SonarCloud.
  8. Editar sonar-project.properties en la raíz del repo y reemplazar ORG y ORG_PROYECTO con esos valores reales.
  9. Commit, push, abrir PR. El job sonarcloud ya correrá.

Cómo leer el dashboard

  • Overview: verde si pasa el gate, rojo si no.
  • Issues: lista de bugs/smells con severidad y archivo.
  • Measures: métricas históricas (cobertura, duplicación, complejidad).
  • Activity: evolución del proyecto en el tiempo.

10. GitHub Actions paso a paso

Anatomía de un workflow

Un workflow es un archivo YAML en .github/workflows/. Estructura general:

name: ci-develop           # nombre que aparece en GitHub UI
on:                        # qué dispara el workflow
  pull_request:
    branches: [develop]
permissions:               # qué permisos tiene GITHUB_TOKEN
  contents: read
jobs:                      # uno o más jobs (corren en paralelo por defecto)
  mi-job:
    runs-on: ubuntu-latest # tipo de máquina virtual
    steps:                 # pasos secuenciales dentro del job
      - uses: actions/checkout@v4   # acción reutilizable
      - run: echo hola              # comando shell

Elementos clave

ConceptoQué es
TriggerEvento que dispara el workflow: push, pull_request, schedule, workflow_dispatch (manual).
JobUnidad que se ejecuta en una máquina aislada. Los jobs pueden depender uno de otro con needs:.
StepPaso dentro de un job. Puede ser un comando (run:) o una acción reutilizable (uses:).
RunnerLa máquina que ejecuta el job. ubuntu-latest es gratuito y lo más usado.
ServiceContenedor auxiliar (ej. Postgres) que arranca antes del job y muere al final.
SecretVariable cifrada (secrets.SONAR_TOKEN). No se muestra en logs.
ArtifactArchivo que un job sube y otro descarga (ej. reportes de cobertura).
CacheMecanismo para reusar node_modules o ~/.cache/pip entre runs.

Pipeline de develop (rápido)

Cuatro jobs en paralelo. Lint backend, lint frontend, pytest, build frontend. Sin gate de cobertura.

name: ci-develop
on:
  pull_request:
    branches: [develop]
jobs:
  lint-backend:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12.10"
          cache: pip
          cache-dependency-path: backend/requirements.txt
      - run: pip install -r requirements.txt
        working-directory: backend
      - run: ruff check app
        working-directory: backend
  # ... (lint-frontend, test-backend, build-frontend) ...

Pipeline de master (estricto)

Cinco jobs. Lint, cobertura backend ≥80 %, cobertura frontend ≥80 %, SonarCloud Quality Gate, build Docker.

El job de Sonar depende de los dos de cobertura (necesita los reportes):

sonarcloud:
  needs: [test-backend-coverage, test-frontend-coverage]
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v4
      with:
        fetch-depth: 0    # Sonar quiere el historial completo
    - uses: actions/download-artifact@v4
      with:
        name: coverage-backend
        path: backend
    - uses: actions/download-artifact@v4
      with:
        name: coverage-frontend
        path: frontend/coverage
    - uses: SonarSource/sonarcloud-github-action@v2
      env:
        GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}

Costo y límites

  • Repos públicos en GitHub: minutos de Actions ilimitados y gratis.
  • Repos privados: 2 000 minutos/mes gratis en cuentas free, 3 000 en Pro.
  • SonarCloud es gratis para proyectos en repos públicos. Pago si tu repo es privado.

11. Branch Protection

¿Para qué sirve?

Sin branch protection cualquiera con permiso de push puede git push --force a master, o mergear PRs aunque el pipeline esté rojo. Branch protection impone reglas a nivel de servidor:

  • Prohibir push directo: todos los cambios pasan por PR.
  • Requerir que ciertos checks de CI estén verdes antes de mergear.
  • Requerir N revisiones aprobatorias (1 en este lab).
  • Prohibir --force-push y eliminación de la rama.

Configuración paso a paso

  1. En GitHub, ir al repo → Settings → Branches.
  2. Click en Add branch protection rule.
  3. Branch name pattern: develop.
  4. Marcar Require a pull request before merging → Require approvals: 1.
  5. Marcar Require status checks to pass before merging. Buscar y agregar:
    • lint-backend
    • lint-frontend
    • test-backend
    • build-frontend
  6. Marcar Do not allow bypassing the above settings.
  7. Save changes.
  8. Crear OTRA regla idéntica para master, pero con los checks de ci-master.yml: lint, test-backend-coverage, test-frontend-coverage, sonarcloud, docker-build.
Los checks solo aparecen en la lista después de que han corrido al menos una vez. Si no los ves, abre primero un PR cualquiera para que GitHub los descubra, luego vuelve a configurar branch protection.

12. Recorrido de las tres etapas

ETAPA 1 Setup inicial

13. Problemas comunes

Setup

python --version sigue mostrando 3.14 después del script Cierra TODAS las terminales abiertas y abre una nueva. PowerShell cachea PATH al inicio. Si VSCode tiene una terminal integrada, reinicia VSCode.
El script no encuentra winget Abre la Microsoft Store y actualiza "App Installer". Después reinicia la terminal.
node dice "command not found" tras instalar con nvm Ejecuta nvm use 20.11.1 nuevamente y cierra/abre terminal. El symlink puede tardar un instante.

Docker

Error "port already allocated" Algo está usando el puerto 5173, 8000, 5432 o 5050. Identifícalo con netstat -ano | findstr 5432 y mátalo, o cambia el puerto en .env.
Error "Cannot connect to Docker daemon" Docker Desktop no está corriendo. Ábrelo desde el menú inicio y espera a que el ícono se ponga verde.
El backend no se conecta a la BD Revisar docker compose logs backend. Lo más común: la BD aún no terminó de inicializarse. El healthcheck debería evitarlo pero a veces toma 30s. Esperar y reintentar.

Tests

pytest dice "no module named app" Estás corriéndolo fuera de backend/. Hazle cd primero. O usa PYTHONPATH=. pytest.
Vitest se queda colgado en "transforming" Limpia: rm -rf node_modules .vite && npm ci.

GitHub Actions

El workflow no se dispara al abrir el PR Verifica que el archivo esté en .github/workflows/ (con punto delante), nombre .yml (no .yaml), y que esté pusheado a la rama base. GitHub solo dispara workflows que existan en la rama destino.
SonarCloud falla con "Not authorized" El SONAR_TOKEN no está bien configurado. Regenera en SonarCloud y vuelve a agregar como secret en GitHub.
Sonar dice "fetch-depth: 0 required" En el step de checkout añade with: { fetch-depth: 0 }.
